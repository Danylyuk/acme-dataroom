## 0. Контекст: чому цей тест — метч

Data Room — це по суті захищений файловий менеджер: дерево папок, вкладеність, upload,
перегляд, rename, delete, обробка конфліктів імен. Я цей шлях уже пройшов у бойових умовах:

- **DeepCloud** (deepcloud.com.ua) — комерційна SaaS хмара на web/iOS/Android, де я соло
  спроєктував і зібрав саме такий складний файл-менеджмент-UI поверх Pydio Cells.

Тобто тут я не «вивчаю задачу з нуля» — я приношу мʼязову памʼять реального продукту.
Що дає бонус: у мене є **власний сервер + домен**, тож деплой піде не на Vercel, а на
власну інфраструктуру під субдоменом — про це нижче.

## 2. Стек і чому саме він

- **Vite + React 18 + TypeScript** — не CRA (він мертвий). Vite = миттєвий dev-сервер.
- **Tailwind + shadcn/ui + Radix** — 1-в-1 стек замовника. Свідомо, щоб зайти «своїм».
- **TanStack Query** — навіть без реального бекенду обгортаю всі операції в
  `useQuery`/`useMutation` поверх локального сховища. Це дає справжній «API-шар»,
  який завтра тривіально підмінити на NestJS-бекенд. Показує володіння їхнім патерном.
- **Dexie.js (IndexedDB)** — єдине сховище і для метаданих, і для PDF-блобів.
  localStorage відпадає (ліміт ~5MB, бінарники не тягне).
- **Google Identity Services** — реальний Google-логін, повністю клієнтський,
  без бекенду (SPA отримує `id_token` + профіль прямо в браузері).
- **react-pdf / iframe + object URL** — перегляд PDF без сервера.
- **react-dropzone, Zod** — drag-drop upload і валідація.

## 3. Модель даних

Дерево зберігаю **плоским списком вузлів** (не вкладеним JSON) — так rename/move/delete
стають тривіальними (міняєш `parentId`), а дерево збираю в памʼяті при рендері.

```
Dataroom { id, name, createdAt, updatedAt }

Node {
  id: string             // uuid
  dataroomId: string     // до якого сейфу належить
  parentId: string|null  // null = корінь Data Room
  type: 'folder' | 'file'
  name: string
  createdAt, updatedAt
  // тільки для file:
  size, mime, blobKey    // посилання на Blob в IndexedDB
}
```

Рекурсивний delete = знайти всіх нащадків по `parentId` і знести разом з блобами.
Це рівно та модель, що працює у DeepCloud поверх Pydio Cells — тому вона мені звична.

**Два рівні застосунку** (важливе прочитання ТЗ — там «create Datarooms», множина):
1. Список Data Room-ів (сейфів) — головна.
2. Всередині сейфу — класичний файловий менеджер з деревом папок.

Більшість кандидатів зроблять одну купу файлів — ми робимо повноцінні ізольовані сейфи.

## 4.2 Безпека: справжнє шифрування at-rest 🔐

«Захищене сховище» тут — не лише слоган. Будь-який Data Room можна **захистити
паролем**, і тоді його файли реально шифруються, а не просто ховаються за логіном.

**Що реалізовано (`src/db/crypto.ts`, Web Crypto / SubtleCrypto):**
- **AES-256-GCM** шифрування вмісту файлів прямо в браузері перед записом у IndexedDB.
- Схема **DEK/KEK** (як у дорослих сховищах):
  `пароль --PBKDF2(210k, SHA-256, random salt)--> KEK --AES-GCM--> розгортає wrappedDEK --> DEK`.
- **Пароль і ключ (DEK) НІДЕ не зберігаються.** У IndexedDB лежить лише `wrappedDEK`
  + `salt`. Розгорнутий ключ живе тільки в памʼяті сесії й зникає при перезавантаженні —
  тоді потрібен пароль знову (екран Unlock).
- Невірний пароль відсікається криптографічно (AES-GCM authentication tag), а не
  порівнянням рядків.
- Зняти захист можна лише ввівши пароль → файли розшифровуються назад.

**Перевірити чесність:** відкрийте DevTools → Application → IndexedDB → `blobs`
для зашифрованого сейфу — там шифротекст, а не ваш PDF.

**Чого свідомо немає (бо це клієнтський MVP без бекенду) — план на продакшн:**
- Серверна автентифікація + сесії (перевірка Google `id_token`, httpOnly-cookie).
- Приватний blob-storage і віддача через **підписані URL** з TTL.
- Рольовий доступ (ACL) до кожного Data Room + аудит-лог доступу.
- Watermark на перегляді, заборона завантаження для read-only ролей.

Архітектура вже готова до цього: UI ходить лише через `hooks/queries.ts` →
`db/api.ts`, тож заміна Dexie на захищений бекенд не зачіпає жоден компонент.

## 4.3 Реальна автентифікація (бекенд) 🔑

Замовник окремо заохотив ініціативу («additions an engineer sees fit»), тож понад
базове ТЗ я додав **справжній auth-бекенд** (папка `server/`) — не мок:

- **Реєстрація email + пароль** з підтвердженням пошти: сервер генерує 6-значний
  код і надсилає його листом через **Resend**, акаунт активується лише після вводу коду.
- **Cloudflare Turnstile** — захист від ботів; токен віджета **валідується на сервері**
  (не клієнтський театр).
- Паролі хешуються **bcrypt**, сесія — **JWT** (7 днів). Дані — **SQLite** (self-contained,
  жодних зовнішніх БД для запуску).
- Захист: ліміт спроб коду, TTL коду 15 хв, не розкриваємо існування акаунта на `resend`.

Стек бекенду: **Express + better-sqlite3 + bcrypt + jsonwebtoken + zod + Resend**.
Ендпоінти: `POST /api/auth/{register,verify,login,resend}`.

Google-логін і email-реєстрація співіснують; демо-режим прибрано (тепер вхід реальний).
Якщо бекенд/ключі не налаштовані локально — код підтвердження друкується в консоль
сервера, а Turnstile використовує тестовий ключ, що завжди проходить (dev-зручність).

## 5. Структура проєкту

```
src/
  db/
    types.ts       # доменні типи (Dataroom, TreeNode, FileBlob)
    db.ts          # Dexie: 3 таблиці + індекси
    api.ts         # "API-шар": весь CRUD, конфлікти імен, рекурсивний delete, move
  hooks/
    queries.ts     # TanStack Query поверх api.ts (єдина точка входу для UI)
  auth/
    AuthContext.tsx  # localStorage + декодування Google JWT
    LoginPage.tsx    # split-screen вхід (Google + демо-fallback)
  components/
    ui/            # shadcn-примітиви (button, dialog, dropdown, card, …)
    AppShell.tsx   # графітовий сайдбар/топбар + меню користувача
    PromptDialog.tsx / ConfirmDialog.tsx / EmptyState.tsx
  pages/
    DataroomsPage.tsx   # список сейфів
    DataroomPage.tsx    # файловий менеджер (breadcrumbs, upload, пошук, view toggle)
    NodeTile.tsx        # плитка папки/файлу з drag-and-drop
    PdfViewerDialog.tsx # перегляд PDF через object URL
  router.tsx       # TanStack Router (code-based) + auth-гейт
  main.tsx         # провайдери
```

**Ключове архітектурне правило:** UI ніколи не звертається до Dexie напряму —
тільки через `hooks/queries.ts` → `db/api.ts`. Тому весь storage-шар замінюється
на реальний бекенд (NestJS/Prisma) без жодної зміни в компонентах.

### Один баг, який варто згадати (чесність)

Перша версія зберігала корінь дерева як `parentId: null`. На аплоаді впало:
IndexedDB **не індексує `null` як ключ**, тож кореневі вузли просто зникали з
складеного індексу `[dataroomId+parentId]`. Рішення: корінь = `parentId === dataroomId`
(завжди валідний рядковий ключ), а `null` лишається тільки в публічному API як
«зручний» вхід і конвертується на межі. Знайшов це живим тестом у браузері, не на очі.

## 6. AI-workflow — де і як я користувався AI

Чесно, бо ви про це просили окремо.

**Інструменти:** Claude Code як агент-виконавець; я вів розробку як живий діалог —
проєктував рішення, агент писав код, я рецензував кожен крок і правив курс.

**Що робив AI:**
- Скафолд Vite+TS, генерація shadcn-примітивів, рутинний boilerplate (dialog/dropdown).
- Первинна реалізація `api.ts` і React Query хуків за моєю специфікацією моделі даних.
- Верстка екранів за моїм описом UX (split-screen логін, картки, breadcrumbs).

**Де я втручався / відхиляв пропозиції AI:**
- **Модель даних — моє рішення, не AI.** Плоский список вузлів + `parentId`, корінь =
  `dataroomId` — це з мого досвіду DeepCloud, а не дефолт агента (він тягнув вкладений JSON).
- **Інтерпретація ТЗ «Datarooms» як множини** (два рівні: список сейфів → файли всередині) —
  моє прочитання; агент за замовчуванням робив один плоский drive.
- **Дизайн-напрям** (графіт + amber, «преміальний сейф, не будівельний жовто-чорний») —
  моє продуктове рішення під конкретного замовника.
- **Баг з `null`-ключем IndexedDB** знайшов і діагностував я через живий тест у браузері;
  агент згенерив початковий (робочий на вигляд, але хибний) варіант.

Тобто AI тримає темп, але архітектурні й продуктові рішення — мої, і я володію кожним рядком.

## 7. Як запустити

**Фронтенд:**

```bash
npm install
npm run dev        # http://localhost:3000
```

**Бекенд (auth):**

```bash
cd server
npm install
cp .env.example .env   # за бажанням заповнити ключі; без них працює dev-режим
npm run dev            # http://localhost:8787
```

Без ключів усе одно працює: код підтвердження **друкується в консоль сервера**,
Turnstile використовує тестовий ключ, що завжди проходить. З ключами —
листи йдуть через Resend, Turnstile перевіряється по-справжньому.

Збірка фронта:

```bash
npm run build      # tsc + vite build → dist/
npm run preview    # локальний перегляд прод-збірки
```

### Змінні середовища

Фронт (`.env` у корені, усі опційні):

```
VITE_API_URL=http://localhost:8787              # бекенд auth
VITE_GOOGLE_CLIENT_ID=<OAuth Client ID>         # реальний Google-логін
VITE_TURNSTILE_SITE_KEY=<Turnstile site key>    # захист від ботів
```

Бекенд — див. `server/.env.example` (`RESEND_API_KEY`, `TURNSTILE_SECRET`,
`JWT_SECRET`, `MAIL_FROM`, `CORS_ORIGIN`).
